# Build the spec-status CLI

Spec overview: docs/specs/2026-04-29-spec-status-view/_overview.md
Spec task:     docs/specs/2026-04-29-spec-status-view/1-ac32-build-cli.md

## Outcome
A new read-only CLI named `spec` ships with the Socrates plugin
as a Nix-built binary. After dev-shell reload, the command
resolves on `PATH` for the human author and for any bash
invocation. Markdown remains canonical; the CLI never mutates
files and never persists derived state.

The CLI must support exactly the following surface (gaps are
intentionally not a subcommand ‚Äî `ls docs/gaps/` is the
existing way to see open gaps):

- `spec` ‚Äî no args. Prints a usage banner listing the
  subcommands and their flags. Exits 0.
- `spec status` ‚Äî lists every non-archived spec under
  `docs/specs/` with: spec name, current phase
  (`Describe` / `Diagnose` / `Delimit` / `Direction` /
  `Design`, derived from phase markers and the
  `delimit_approved:` frontmatter), and per-status task
  counts.
- `spec tasks` ‚Äî lists tasks across all non-archived specs
  with one row per task, including the ticket id from the
  task's `ticket:` frontmatter (set by `/pour`; `-` for
  unpoured tasks). Accepts:
  - `--status STATUS` ‚Äî filter by `status:` frontmatter value
    (`draft` / `approved` / `poured` / `cancelled`).
  - `--review` ‚Äî show only tasks whose `<review>...</review>`
    block contains non-whitespace content.

Output is plain human-readable text. No `--json` flag in this
spec.

### Expected output examples

Drafts based on the project's current state. Exact column
widths, separators, and ordering are the implementer's call;
the *contract* is each example's columns and per-row
information content.

`spec` (usage):

```
Usage: spec <command> [options]

Commands:
  status              List non-archived specs with phase and task counts
  tasks               List tasks across non-archived specs
  tasks --status S    Filter tasks by status (draft|approved|poured|cancelled)
  tasks --review      Show only tasks with non-empty <review> blocks
```

`spec status`:

```
2026-04-29-pr-review-loop
  Phase:    Design [COMPLETE]
  Delimit:  approved
  Tasks:    2 draft, 0 approved, 0 poured

2026-04-29-spec-status-view
  Phase:    Design [COMPLETE]
  Delimit:  approved
  Tasks:    1 draft, 0 approved, 0 poured
```

`spec tasks`:

```
SPEC                          TASK                              STATUS  PRI  CATEGORY    TICKET     REVIEW
2026-04-29-pr-review-loop     1-13a3-implement-review-mode      draft   0    functional  -          -
2026-04-29-pr-review-loop     2-e75b-init-stamp-review-mode     draft   1    functional  -          -
2026-04-29-spec-status-view   1-ac32-build-cli                  draft   0    functional  -          -
```

Columns required: SPEC, TASK, STATUS, PRI, CATEGORY, TICKET,
REVIEW. The `TICKET` cell shows the value of the task's
`ticket:` frontmatter when set (e.g., `rmg-001`) and `-`
otherwise. The `REVIEW` cell shows `pending` when the task's
`<review>` block has non-whitespace content and `-` otherwise.

`spec tasks --review`:


### Packaging and quality gating

- **Nix-built binary.** A new Nix package builds the script
  using `pkgs.writeShellApplication` (or equivalent), placed
  consistently with the project's existing nixDir conventions.
  `writeShellApplication` runs shellcheck during the build;
  any diagnostic fails the Nix build.
- **Source script in plugin templates.** The script source
  lives at `plugins/socrates/templates/spec` (no extension,
  matching the binary name). The Nix package builds from this
  source.
- **Pre-commit shellcheck hook (Nix-managed).** The project's
  `.pre-commit-config.yaml` is a read-only Nix-store symlink
  generated by `git-hooks.nix`. The new shellcheck hook is
  added in `nix/devenvs/default.nix` via
  `git-hooks.hooks.shellcheck = { enable = true; ... };`
  scoped to bash files under `plugins/socrates/templates/`.
  Defense-in-depth alongside the build-time check: pre-commit
  catches issues before push, Nix build catches them before
  deploy.

### Implementation guidance

Bash + `jq` is the established plugin language. Frontmatter
parsing is well within bash + small helpers. The trickiest
piece is `<review>`-block extraction across multi-line
Markdown body ‚Äî the existing scripts in
`plugins/socrates/templates/` parse only frontmatter, not body
blocks, so this is new ground. The implementer may reach for
`awk`/`sed` or `yq` if it keeps the script tractable.

Tests follow the existing harness pattern (see
`plugins/socrates/templates/spec-read-guard.test.sh`): a
sibling `*.test.sh` file with a `run()` helper, exit 0 = pass,
exit 1 = fail. Tests must use a fixture directory tree (e.g.,
under `mktemp -d`) ‚Äî the real `docs/specs/` and `.tickets/`
trees are not safe to use as test inputs because tests should
not depend on project state. The fixture must exercise: a
fresh spec at `Describe [DRAFT]` with no tasks; a spec
mid-phase (`[APPROVED]` Delimit, `[DRAFT]` Direction); a
poured spec with `epic:` set and a task with a populated
`ticket:` field; a task with a non-whitespace `<review>`
block; a task with an empty `<review></review>`.

## Verification
- `nix build` of the new package succeeds; the resulting store
  path contains a `spec` binary under `bin/`.
- After dev-shell reload, `which spec` resolves and `spec`
  prints a usage banner with the columns shown in the
  `spec` (usage) example, and exits 0.
- Running `spec status` from the repo root with the project's
  current state lists at least the
  `2026-04-29-pr-review-loop` and `2026-04-29-spec-status-view`
  specs with their phases, `delimit_approved` state, and
  per-status task counts.
- Running `spec tasks` against a fixture containing one poured
  task (with `ticket: SENTINEL-1` in its frontmatter) and one
  unpoured task shows `SENTINEL-1` in the TICKET column for
  the first row and `-` for the second.
- Running `spec tasks --status approved` against a fixture
  with tasks at multiple statuses returns only the approved
  ones.
- Running `spec tasks --review` against a fixture with two
  tasks (one with non-whitespace `<review>` content, one
  empty) returns only the first.
- The sibling `*.test.sh` exists, executes against the
  fixture tree, and exits 0.
- Introducing a known shellcheck violation (e.g., unquoted
  `$VAR` in a context that needs quoting) into the script and
  running `nix build` fails with a shellcheck diagnostic;
  reverting passes again. (One-time validation by the
  implementer; not a durable test.)
- Running `pre-commit run shellcheck --files
  plugins/socrates/templates/spec` succeeds on the clean
  script and fails when the same violation is reintroduced.
- The script is read-only by construction: a `grep` over the
  script for `>` (output redirection to file paths under
  `docs/`), `sed -i`, `>>`, or any `tk` mutation subcommand
  returns nothing.
